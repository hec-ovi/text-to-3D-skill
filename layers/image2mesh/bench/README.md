# Benchmark method

Every performance claim in this repo comes from `run-bench.sh` on the box described below. Numbers live in `results/*.json`; the summary table is in [`../CHANGES.md`](../CHANGES.md).

## Box

| | |
| --- | --- |
| APU | AMD Ryzen AI Max+ (Strix Halo), Radeon 8060S, gfx1151 |
| Memory | 128 GB unified, GTT carries the weights, VRAM stays a small BIOS carve-out |
| Driver | Mesa RADV 26.0.3 |
| Host | Ubuntu 26.04, kernel 7.0 |
| Container | `ubuntu:26.04`, `/dev/dri` only, render and video groups added, no `--privileged` |

## What is held fixed

- Same input image (`../fixtures/bench-subject.png`, generated by the text2image layer at seed 42, so it is reproducible without a checked-in binary asset from anywhere else).
- Same engine seed (42), same resolution, same flags including `--require-gpu`.
- ComfyUI stopped, so the 15 GB it keeps resident is not competing for bandwidth. Start it again before running the pipeline end to end.
- Page cache warm: the weights have been read at least once before the timed run. A cold-cache first run pays disk on top and is not what the table reports.

## What is recorded

| Field | Meaning |
| --- | --- |
| `wallSeconds` | `docker run` to exit, so container startup and Vulkan device init are inside it. This is the number a caller feels. |
| `engineSeconds` | The engine's own `done in Ns`, which excludes process startup. |
| `phases[]` | Seconds between consecutive `[n/m]` banners. The engine keeps stdout unbuffered, so stamping each line attributes time per stage without touching the C++. The first entry covers process start plus the first model load. |
| `peakGttMiB`, `peakVramMiB` | Highest reading of the kernel amdgpu counters under `/sys/class/drm/card*/device/` during the run. |
| `gttDeltaMiB`, `vramDeltaMiB` | Peak minus the reading taken just before the run. These counters are system-wide, so the delta is the figure to compare between tags. |
| `triangles`, `glbBytes` | Read out of the produced GLB. A change that gets faster by producing a smaller mesh is not a speedup, and this is how that shows. |

## Running it

```bash
# before: pinned upstream, Vulkan backend, none of our changes
./run-bench.sh --tag baseline --image ../fixtures/bench-subject.png \
  --docker-image text-to-3d/engine:baseline --entry "" --res 512

# after: this repo's engine
./run-bench.sh --tag trimmed --image ../fixtures/bench-subject.png \
  --docker-image text-to-3d/engine:vulkan --res 512
```

`--entry ""` exists because the baseline image's entrypoint is the binary itself, while ours takes a `cli` or `server` subcommand first.

Single runs, not averages: at six-plus minutes each, a rep count large enough to be statistically interesting costs more than the decisions it informs. Treat differences under about 5% as noise.
